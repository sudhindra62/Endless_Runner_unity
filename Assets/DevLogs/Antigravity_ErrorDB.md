## BLOCK 09 🧬 ERROR VARIETY DETECTOR (CAUSE-LEVEL INTELLIGENCE)

### CORE LAW
**Same ErrorType ≠ Same Error.**
Do NOT classify by CS code alone. Classify by: RootCause + Context + System.
Every unique root cause gets its own `[AG_ERR_xxx]` entry — even if CS code matches.

### CLASSIFICATION DIMENSIONS (use all 3)
| Dimension | What to identify |
|-----------|------------------|
| **Root Cause** | WHY did it fail? (file missing, namespace wrong, method absent, enum incomplete, brace dropped...) |
| **Context** | WHERE in the architecture? (Manager, DataModel, UI, Event, Enum, Config, Gameplay...) |
| **System** | WHICH game system? (Theme, Skin, Combat, Economy, AudioSystem, LiveOps, Integrity...) |

### MULTI-CAUSE EXAMPLE (CS1061)
```
CS1061 — Pattern A: Missing method on class        → class exists, method not declared
CS1061 — Pattern B: Method on wrong class entirely → calling ManagerA instead of ManagerB
CS1061 — Pattern C: Namespace mismatch             → method exists but class not in scope
CS1061 — Pattern D: Missing property on data model → field not declared on SaveData/AchievementData etc.
CS1061 — Pattern E: Static called as instance      → static method called via instance ref or vice versa
```
All 5 are DIFFERENT entries in ERROR_REGISTRY. Never compress into one.

### MULTI-CAUSE EXAMPLE (CS0246)
```
CS0246 — Pattern A: File physically deleted         → class file does not exist
CS0246 — Pattern B: Namespace trapped               → class exists but inside subnamespace
CS0246 — Pattern C: Post-hoist stale qualifier      → class hoisted but old full path still used
CS0246 — Pattern D: Wrong assumed class name        → coded wrong name (RadialTimer vs PowerUpRadialTimer)
CS0246 — Pattern E: Companion class not created     → primary exists but linked class never made
CS0246 — Pattern F: Enum not declared               → enum type used but no .cs file for it
```
Each = different registry entry.

### DETECTION FLOWCHART
```
Error received
    │
    ├─► Identify ErrorType (CS code)
    ├─► Identify Root Cause (WHY)
    ├─► Identify Context (WHERE in arch)
    ├─► Identify System (WHICH feature)
    │
    ├─► Search registry for matching Cause text
    │      ├─► EXACT match found → Count+1, update Seen
    │      └─► No match → NEW entry (even if CS code exists)
    │
    └─► Apply matching PREVENTION_RULE before fixing
```

## BLOCK 11 🟠 PATTERNS

[P01] NamespaceTrapping — Class inside `namespace EndlessRunner.Sub` → invisible globally. Always check scope before assuming accessibility.
[P02] DeletedManagerVoid — Merge/AI deletes manager file → 100+ cascade errors. Stub-first: create empty Singleton before any consumer.
[P03] StaticVsInstanceEventMix — Event declared instance but subscribed statically. Rule: if subscriber uses ClassName.Event → static.
[P04] DuplicateByStacking — Method injected without reading file first → duplicate signature. Read full file + grep name before injecting.
[P05] PostHoistStaleReference — After namespace strip: callers still use full-qualified name OR consuming `using` still imports deleted namespace.
[P06] SystemFullDesync — Architectural updates or AI file overwrites strip expected properties/methods from singletons globally, compounding into mass compiler failures.
[P07] EnumValueMissing — Enum exists but lacking specific value → cascade errors. When adding feature: enumerate ALL required values at once.
[P08] DataModelIncomplete — Class exists but fields UI needs are absent. Write fields for a data model BEFORE writing consumers.
[P09] DuplicateFilesSameClass — Two .cs files same class name → ambiguous. grep BEFORE create. Merge, never duplicate.
[P10] CompanionClassOrphaned — Primary class created but companion (Config, Data, Type) not created simultaneously.
[P11] StringVsEnum — Raw string passed where enum expected after signature change. Any category distinction = typed enum.
[P12] ReadOnlyDirectAssign — get-only property assigned directly. Expose SetXxx() method for external mutation.
[P13] SingleSourceOfTruth — Same logic (currency, state) in 2+ managers. Designate one authority; others delegate.
[P14] APINameCasing — Unity API misspelled (targetFramerate vs targetFrameRate). Verify casing from docs, never from memory.
[P15] LINQImplicitTyping — LINQ lambda on custom type fails without explicit typing. Always type LINQ parameters explicitly.
[P16] HeuristicBraceInjection — Automated scripts (like `tmp_restore_braces.ps1`) inject `{` based on line-endings without parsing state. Results in 100+ CS1519/CS1022 errors. Use AST-aware tools only.
[P17] ToolOutputTruncationRisk — Large-scale project audits (e.g. `grep` or `Get-ChildItem` across 200+ files) may truncate results in the terminal/view tool. This leads to incomplete structural restoration.
[P18] SystemicBranchCollision — Divergent development branches (`Assets/Game` vs `Assets/Scripts`) creating duplicate gameplay systems. Resolve via precision logic merge + total tree purge.
[P19] NestedClassHoisting — Promoting nested types to global scope breaks all `Class.NestedType` references across the codebase. Globalize all enums and data struct models.
[P20] AbstractContractDrift — Renaming/changing signatures in base classes breaks all leaf implementations. Verify leaf classes when updating interfaces/abstracts.

## BLOCK 12 🟡 PREVENTION_RULES

[R01] No undefined type before compile → Every referenced .cs class must physically exist before the consumer compiles.
[R02] Stub-first for managers → Empty Singleton<T> before any UI/gameplay that references the manager.
[R03] No string for typed category → CurrencyType, RewardType, AchievementID, PowerUpType, CosmeticRarity = always enum.
[R04] Global scope for core systems → ObjectPool, ServiceLocator, GameState, GameEvents, all Singleton managers = no namespace.
[R05] Read file before injecting → Grep method name in file. Only add if absent. Never assume.
[R06] Verify brace balance post-edit → View file after every namespace-strip or multi-replace on class wrapper.
[R07] Post-hoist grep cleanup → After any namespace removal: grep `EndlessRunner.SubNS.ClassName` → strip all occurrences.
[R08] One type per name per project → Duplicate class/struct name = guaranteed CS0101/CS0436. grep before create.
[R09] Data model first → ScriptableObject + serializable classes created BEFORE any manager or UI consuming them.
[R10] Static events if subscribed by class name → Instance events only when instance reference always available.
[R11] Add ALL enum values at once → When a feature needs enum values, add every value the feature touches in one pass.
[R12] Companion classes in same commit → ThemeSO+ThemeConfig, CharacterSkin+SkinUnlockType = same file or same edit.
[R13] Unity API casing from docs → Never trust memory for Unity API names. grep working usages in project.
[R14] Single currency authority → Only PlayerDataManager touches coins/gems. All others call through it.
[R15] NamespaceStubs.cs audit on every edit → Remove stub entry the instant a real class file is created for it.
[R16] Delete atomically → Deleting a class: grep all event subscriptions, method calls, using directives → reroute all before delete.
[R17] Caller-callee sync → Method changed → grep all callers → update all in same edit. Never leave stale callers.
[R18] Float→int is never implicit → Explicit (int) cast required at every float-to-int boundary crossing in C#.
[R19] Explicit LINQ typing → Lambda parameters over custom types must be explicitly typed, not var.
[R20] Config fields declared before pipeline → LiveOpsConfigProfile / RemoteConfig fields must exist before the pipeline that reads them.
[R21] NO BULK STRUCTURAL REPLACEMENT → Never use automated scripts to delete structural tokens (`{`, `}`) across multiple files. All architectural modifications MUST be performed via precision high-confidence edits or AST-aware automation.
[R22] AUDIT COMPLETENESS LAW → For project-wide structural fixes, never rely on terminal snapshots. ALL audits MUST output to a persistent file (e.g. `audit_results.txt`) and be verified via total character count contrast before declaring a mission complete.
[R23] Data models and Enums are standalone → Never nest data definitions inside Manager classes, avoiding CS0426 nesting errors.
[R24] Manager entry-points are public → Any method designed to be called from a coordinator or external script MUST be declared `public`. Default to `public` for all manager API methods; use `private` only for helpers.
[R25] READ callee before writing caller → Before writing any call to a manager method, open the manager file and verify the method name, parameters, and access level exist exactly as expected.
[R26] One canonical enum per domain → When two enums cover same concept (AchievementID vs AchievementType), immediately designate one as the canonical cross-system type and update all callers atomically.
[R27] Domain Bridge Overloading — When upgrading a system to use high-level objects (Quest) while legacay code uses data models (QuestData), provide overloads for BOTH in the consumer API to prevent CS1501/CS1503 errors during transition.
[R28] Model-First API implementation — Data structures and their serializable models MUST be 100% complete before any consuming API or UI logic is written.
[R29] Event Scope Validation — Every event subscription must be checked against its declaration scope (Static vs Instance) to prevent CS0120/CS0176 errors during inter-manager wiring.
[R30] Method-Wrapped Events — No event subscription (`+=` / `-=`) shall be placed directly in class scope. All event logic MUST reside in `OnEnable`, `OnDisable`, `Start`, or a dedicated registration method.
[R31] Overload Provisioning — For any manager method with complex parameters, always provide a simplified or parameterless overload to prevent caller drift and CS1501 errors in the UI/Coordinator layers.
[R32] UnifiedSourceOfTruth — Standardize on a single authoritative data model for persistence. Delete redundant proxy classes (SaveData.cs).
[R33] SerializableDictionarySupport — Use BinaryFormatter for persistence and custom stable string hashing for checksums to ensure dictionary integrity.
[R34] TerminologicalUnification — When a model is purged/renamed, all associated method signatures, variable names, and string keys (PlayerPrefs) MUST be updated to the new terminology project-wide.
[R35] Unified Data Interface — Core data models (SO and Runtime) MUST expose a standardized set of capitalized properties (Name, Description, Price, Amount, TargetValue) to satisfy UI controllers, regardless of internal field naming.
[R36] Editor Assembly Compatibility — Every `Assets/Editor` tool MUST validate package API presence and target-type constraints before invocation. Never call editor/package helpers by memory alone, and never instantiate non-`ScriptableObject` runtime models through `CreateInstance<T>()`.
